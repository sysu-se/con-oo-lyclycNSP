## HW 问题收集

列举在 HW1、HW1.1 过程中，我所遇到的 2~3 个通过自己学习已经解决的问题，和 2~3 个尚未完全解决的问题与挑战。

### 已解决

1. View 层如何消费 `Sudoku` / `Game`？

   1. **上下文**：HW1.1 要求真实 Svelte 前端消费领域对象，而不是只在测试中使用 `Sudoku` / `Game`。一开始我不清楚“View 层”和“消费”具体指什么，容易理解成只要组件能间接拿到一个二维数组就可以。

   2. **解决手段**：结合 `作业要求1.md` 和 Codex Review，我重新梳理了 UI 调用链。这里的 View 层主要是 Svelte 组件和它们直接依赖的 store，例如 `Board/index.svelte`、`Keyboard.svelte`、`Actions.svelte` 和 `stores/grid.js`。“消费”指 View 层把 `Sudoku` / `Game` 当成真实的数据和行为来源：棋盘渲染读取领域对象导出的 `grid` / `invalidCells` / `canUndo` 等状态，用户输入、撤销、重做则进入 `Game.guess()`、`Game.undo()`、`Game.redo()`，而不是直接修改二维数组。

2. 为什么需要 Store Adapter，而不是让组件直接操作领域对象？

   1. **上下文**：`Sudoku` / `Game` 是普通 JS 对象，内部字段变化后，Svelte 并不会自动知道对象已经改变。如果组件直接持有 `Game`，调用 `game.guess()` 后，界面不一定刷新。

   2. **解决手段**：我采用了类似 Store Adapter 的方式，让 Svelte store 内部持有 `Game`，并在每次领域对象变化后调用 `state.set(...)` 或相关 store 的 `set(...)`，把领域对象当前状态投影成 Svelte 可订阅的数据。这样组件通过 `$store` 消费响应式状态，Svelte 能收到通知并重新渲染。

3. 题面固定格为什么应该进入领域模型？

   1. **上下文**：HW1 的实现里，`Sudoku` 主要像一个 9x9 数字矩阵，`guess()` 可以覆盖任意位置。Codex Review 指出这没有建模数独最核心的业务概念：题目初始给定数字不能被玩家修改。

   2. **解决手段**：我理解到 `Sudoku` 不应该只保存当前 `grid`，还应该保存题面或固定格信息，例如 `puzzleGrid` / `givenCells`。这样 `guess()` 可以先判断目标格是否可编辑，UI 也可以根据领域对象导出的题面状态区分“固定数字”和“用户输入数字”。这让规则不再散落在组件或 store 中，而是由领域对象维护。

### 未解决

1. `isValidMove` 和 `guess` 的语义边界还需要进一步明确。

   1. **上下文**：Codex Review 指出当前领域层中 `isValidMove` 与 `guess` 的规则语义有分裂风险。如果 `isValidMove` 表示“数独规则上完全合法”，但 `guess` 仍允许用户填入冲突数字，那么调用方可能误解 API。

   2. **尝试解决手段**：我目前的理解是，数独游戏可以允许玩家填错并在 UI 中高亮冲突，所以 `guess` 不一定要拒绝所有冲突值。但这样就需要更清楚地区分两个概念：`canEdit` / `canPlace` 表示能不能写入该格，`isValidMove` 或 `getInvalidCells` 表示写入后是否违反数独规则。后续还需要把 API 命名和文档说明改得更准确。

2. 新开局时哪些状态应该属于 `Game` 会话还没有完全收拢。

   1. **上下文**：Review 指出 `startNew` / `startCustom` 已经会创建新的 `Game`，但候选数 `candidates` 和笔记模式 `notes` 仍然游离在外部 store 中。如果新开一局时没有清空候选数或重置 notes，上一局的 UI 状态可能泄漏到下一局。

   2. **尝试解决手段**：我已经意识到一局游戏不只包含棋盘二维数组，还包括一些会影响输入语义和显示效果的会话状态。后续需要决定这些状态是否应该纳入 `Game`，或者至少由统一的游戏启动流程集中 reset。目前还没有完全确定最合适的边界。

3. 提示功能和领域规则之间的关系仍有设计挑战。

   1. **上下文**：`applyHint` 会对当前盘面调用外部 solver，再把求出的数字写回 `Game`。但如果玩家当前已经填出了冲突盘面，当前盘面可能不可解，提示功能就依赖第三方 solver 的容错，而不是由领域层明确保证前置条件。

   2. **尝试解决手段**：我认为提示流程前应该先通过领域对象检查当前局面是否存在冲突，或者改成基于原始题面解答来给提示，而不是直接求解被玩家修改后的盘面。这个问题涉及 `Sudoku` 是否要保存标准答案、hint 是否属于领域能力等问题，目前还没有完全解决。
