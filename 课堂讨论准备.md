# 课堂讨论准备

## 基于 Homework 1.1 的问题

### 1. 你的 view 层直接消费的是谁？

view 层直接消费的是一个面向 Svelte 的 adapter store，而不是直接消费 `Game` 实例本身。  
具体位置在 `src/node_modules/@sudoku/stores/grid.js`。

这个 adapter 在闭包里持有真正的 `Game` 领域对象，然后向组件暴露两类内容：

- 可订阅状态：`grid`、`userGrid`、`invalidCells`、`canUndo`、`canRedo`、`won`
- 可调用命令：`generate`、`decodeSencode`、`set`、`applyHint`、`undo`、`redo`

这样做的原因是：领域层保持框架无关，Svelte 只消费它需要的响应式视图状态。

### 2. 为什么你的 UI 在领域对象变化后会刷新？

因为领域对象变化后，我没有指望 Svelte 自动追踪普通对象字段，而是显式调用了 store 的 `set(...)`。

流程是：

1. 组件事件调用 adapter 暴露的方法
2. adapter 内部调用 `currentGame.guess()` / `undo()` / `redo()`
3. adapter 再执行 `state.set(currentGame.getState())`
4. Svelte 检测到 store 被重新赋值，刷新所有依赖 `$store` 的组件

所以 UI 刷新的根本原因不是“对象内部变了”，而是“store 重新发布了新的纯数据状态”。

### 3. 你的方案中，响应式边界在哪里？

响应式边界在 Svelte adapter store。

- 领域对象内部：`Sudoku` / `Game` 自己管理业务状态，但它们本身不是 Svelte store
- 响应式边界：`grid.js` 中的 `state` writable store
- View 层：通过 `$grid`、`$userGrid`、`$gameCanUndo` 等订阅这个边界之外的纯数据

也就是说，领域层负责“状态和规则”，adapter 负责“把状态投影给 Svelte”。

### 4. 你的 `Sudoku` / `Game` 哪些状态对 UI 可见，哪些不可见？

对 UI 可见的状态主要是：

- `puzzleGrid`
- `grid`
- `invalidCells`
- `won`
- `canUndo`
- `canRedo`

这些状态通过 `Game.getState()` 暴露给 adapter，再由 adapter 暴露给组件。

对 UI 不可见、只保留在领域对象内部的状态主要是：

- `Sudoku` 内部保存的真实二维数组引用
- `Game` 的 `undoHistory` / `redoHistory` 的具体快照结构
- 固定格保护和冲突判断的内部实现细节

UI 只能看到结果，不直接操纵这些内部结构。

### 5. 如果将来迁移到 Svelte 5，你的哪一层最稳定，哪一层最可能改动？

最稳定的是领域层，也就是 `src/domain/index.js` 里的 `Sudoku` / `Game`。

原因是：

- 它们不依赖 Svelte API
- 它们处理的是业务语义、历史、序列化和校验
- 这些需求在 Svelte 3 或 Svelte 5 下都不会本质改变

最可能改动的是 adapter 层，也就是 `src/node_modules/@sudoku/stores/grid.js`。

原因是：

- 它直接依赖当前 Svelte 3 的 store 机制
- 如果换成 Svelte 5 的 runes 或其他响应式写法，首先需要重写的就是这层“领域对象 -> 视图状态”的桥接方式

换句话说，迁移框架时我希望改的是适配层，而不是领域层。

## 基于 Homework 1 的问题

### 1. 除了 `Sudoku` 和 `Game` 两个对象之外，是否还应该有其它对象？

可以有，但不一定都要提升为核心领域对象。

例如：

- `Move`：适合作为轻量值对象
- `GameStore` / adapter：适合作为应用层对象，而不是领域对象
- `HintService`：如果以后提示逻辑变复杂，可以单独抽服务对象

就当前作业规模而言，`Sudoku` 和 `Game` 足够支撑核心设计，其它对象更多是“为了接入或扩展而增加”，不是必须的核心实体。

### 2. 为什么 `Move` 不是核心领域对象？

因为它没有独立身份，也不需要长期生命周期。

`Move` 只描述一次输入：

```js
{ row, col, value }
```

两个字段相同的 `Move` 在业务上没有区别，因此更像值对象而不是实体对象。  
如果把它做成核心对象，反而会让当前作业的模型变重。

### 3. 你的 Undo/Redo 更接近“存 move”还是“存 snapshot”？

我现在采用的是“存 snapshot”。

历史栈里存的是：

```js
{
  grid,
  puzzleGrid
}
```

这样撤销和重做都更直接：只要恢复整盘状态即可，不需要反推某一步之前的旧值。

### 4. 你的设计中最容易出错的地方是什么？

最容易出错的地方有两个：

1. 深拷贝不彻底
   如果历史快照和当前盘面共享同一份二维数组引用，Undo/Redo 会被污染。

2. 响应式边界不清楚
   如果只改 `currentGame` 内部状态，但忘记调用 store 的 `set(...)`，界面就会出现“数据变了但 UI 不刷新”的问题。

这两个点一个是领域层状态隔离问题，一个是 Svelte 接入问题，都是这次作业最容易失分的地方。

### 5. 如果继续增加“提示”或“探索模式”，你现在的设计够用吗？

对“提示”来说，当前设计是够用的。

原因是：

- `Game.guess(...)` 已经统一了普通输入和提示输入的状态演进
- Undo/Redo 和序列化逻辑不需要重写

对“探索模式”来说，当前设计只能算部分够用。

如果探索模式只是普通临时输入，现有模型还能继续用；  
但如果要支持多分支历史、草稿状态、分叉回溯，那么目前的双栈快照历史就不够了，`Game` 的历史结构需要升级成树形或带分支的结构。
